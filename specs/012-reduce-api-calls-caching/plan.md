# Implementation Plan: Reduce API Calls Per Session & Improve Caching

**Branch**: `012-reduce-api-calls-caching` | **Date**: 2026-04-13 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/012-reduce-api-calls-caching/spec.md`

## Summary

The app makes ~18 API calls on startup (7 to `launches/` alone) due to fragmented ViewModels, wasted preload tasks, and uncached repository methods. This plan:
1. Removes 6 wasted Tier 1 preload tasks and aligns tiers with actual Home consumption
2. Adds stale-while-revalidate caching to 3 uncached `LaunchRepository` methods
3. Defers below-fold Home sections (stats, history, previous launches) until scrolled into view
4. Consolidates overlapping launch queries via a shared HomeDataCoordinator
5. Optionally adds Ktor HTTP-level caching for 304 support

Target: Reduce returning-user Home screen API calls from ~11 to ≤5.

## Technical Context

**Language/Version**: Kotlin 2.0+ (KMP), Java 21 (JetBrains JDK)
**Primary Dependencies**: Ktor (HTTP client), Koin (DI), Jetpack Compose Multiplatform, kotlinx.serialization
**Storage**: SQLDelight (persistent cache), in-memory `LaunchCache`, JSON blob serialization
**Testing**: `commonTest` (shared), `jvmTest` (desktop), `iosTest`, Android instrumentation
**Target Platform**: Android (primary), iOS, Desktop (JVM)
**Project Type**: Mobile (Kotlin Multiplatform)
**Performance Goals**: Home screen first content ≤2s from cache; ≤5 API calls on returning-user mount
**Constraints**: Offline-capable (stale-while-revalidate), low-RAM device awareness, API rate limits
**Scale/Scope**: ~29 ViewModels, 8 SQLDelight cache tables, 7 LocalDataSource classes, 12+ Repository implementations
